# 工作准则 - Sitimo

## 防止大文件写入失败与上下文膨胀

基于 2024-06-06 凌晨因连续 18 次 `Write`/`Edit` 参数丢失导致写入失败、最终上下文膨胀至 131072 token 上限崩溃的故障，制定以下强制规则：

### 1. 大文件写入优先用 Bash heredoc
当写入/编辑的内容超过 100 行或 5 KB 时，**禁止使用 `Write`/`Edit` 工具直接写入大段内容**。改用 Bash：

```bash
cat <<'EOF' > /path/to/file.md
...长内容...
EOF
```

> 原因：模型在生成长文本内容的同时，无法正确填充 `Write`/`Edit` 的 JSON `content` 参数，导致空参数调用全部失败。

### 2. 分块写入，逐块验证
若必须使用 `Write`/`Edit`，将内容拆成 3-5 个独立小块（每块不超过 50 行）。
每块写入成功后，用 `Read` **立即验证落盘**，再继续下一块。
禁止一气发送 5-10 个连续的 `Edit` 追加操作。

### 3. 45 分钟规则 - 主动断点保存
长时间代码探索（大量读文件、搜索、Agent 调用）后，**每隔 30-45 分钟主动保存进度并 `/exit`**，然后开启新 Session 继续。

新 Session 上下文清零，token 压力归零，避免陷入“死亡螺旋”。

### 4. 写入前先做预演
在正式写大文件之前，先执行一条极简的测试：

```bash
echo "test" > /tmp/test-write.txt
```

确认工具调用通道正常，再执行真正的写入操作。

### 5. 连续 2 次失败 = 立即换策略
如果一个 `Write`/`Edit` 调用**连续失败 2 次**，禁止继续第 3-18 次重试。必须：
- 把内容先写到 `/tmp/draft.md`（用 Bash `cat <<'EOF'`）
- 然后用 `Bash` 命令 `mv /tmp/draft.md /final/path.md`
- 或者把内容拆成更小的块

### 6. 上下文膨胀预警
工作期间若出现以下任一信号，**立即保存退出**：
- 模型回复开始变短、重复或格式混乱
- 每次工具调用响应延迟明显增加
- 看到 `API Error`、`token limit`、`context length` 类提示
- 连续 3 次空参数错误或 `Error writing file`

---

以上规则优先级高于所有默认系统行为。任何情况下，持久化用户数据与避免上下文崩溃，优先于任务完成速度。
